## Objective

Build a first-class toolchain API documentation substrate layer that reduces agent context burn and command error rates. Agents should be able to consult a locally-cached, structured reference for heavily-used CLI tools (starting with `gh`) rather than re-deriving command syntax from memory or burning tokens on web fetches.

This is a new foundational layer — it will grow as new tools are added. The `gh` CLI is the first tool because it is used in almost every session and has the highest-documented failure rate (silent body corruption, Projects v2 auth scopes, API vs REST confusion).

## Design Decisions

### Why start with `gh`?
Used in every session. Has the highest-documented failure rate in this repo's memory:
- `--body` multi-line corruption (AGENTS.md guardrail)
- Projects v2 requires separate auth scope
- REST vs GraphQL confusion for project field values
- Silent API failures requiring verify-after-act

### Why `.cache/toolchain/gh/`?
Consistent with existing `.cache/sources/` pattern. Gitignored. Auto-regenerated by script.

### Why `docs/toolchain/`?
Committed docs are visible to agents in context. The `.cache/` layer is the raw machine output; `docs/toolchain/` is the curated, agent-readable synthesis — similar to how `docs/research/sources/` relates to `docs/research/`.

### Toolchain reference encoding in AGENTS.md
A new convention: **"toolchain-reference-first"** — before constructing or suggesting any `gh`, `uv`, `ruff`, or `git` command, agents must check `docs/toolchain/<tool>.md` for the canonical pattern for this repo. This prevents re-deriving known-safe patterns and re-encountering known failure modes.
